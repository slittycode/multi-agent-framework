## Boundary Compatibility Validation

### A) Adapter IDs and synthesis IDs vs provider routing
- Built-in adapters all validate with zero errors.
- Built-in agents define `llm.provider` values compatible with registry resolution (`mock`).
- `synthesisAgentId` references real agent IDs in all built-ins.
- Conclusion: adapter identity model is compatible with provider routing.

### B) `ProviderGenerateResult` -> `Message` -> transcript append
- `ProviderGenerateResult` exposes `content`, `provider`, `model`, `usage`.
- `Message` accepts these fields (`content`, `provider`, `model`, `usage`) plus routing metadata.
- `appendMessage(transcript, message)` contract is sufficient for lossless insertion of provider output.
- Conclusion: no type-level loss across provider output to transcript entry.

### C) Error model consistency for orchestrator handling
- Adapter path currently throws `AdapterLoadError` (with code + details).
- Provider path currently throws registry-specific errors (`UnknownProviderError`, `MissingAgentProviderError`, `DuplicateProviderError`).
- Both are deterministic and typed by class, but not yet normalized under a shared orchestrator error envelope.
- Conclusion: compatible, but normalization adapter is recommended in Step 6.

### D) Transcript persistence contract for JSON + JSONL
- `serializeTranscriptToJson(transcript): string` and `serializeTranscriptToJsonl(transcript): string` locked.
- `persistTranscript({ transcript, outputDir, format }): Promise<{ path }>` locked.
- `format` union already matches orchestrator config (`"json" | "jsonl"`).
- Conclusion: contract supports both formats without type changes.

## Interface Gaps and Remediation

### Gap 1 (Medium): Heterogeneous error surfaces across adapters/providers
- Impact: future orchestrator may require duplicated error branching logic.
- Remediation: introduce `OrchestratorIntegrationError` wrapper in Step 6 with `source`, `code`, `cause` fields and map from loader/registry errors.

### Gap 2 (Low): Built-in resolution heuristic is name-vs-path based
- Impact: unusual adapter IDs containing path-like patterns could be treated as paths.
- Remediation: in Step 6 CLI, reserve explicit `--adapter-id` and `--adapter-file` flags to remove ambiguity.

### Gap 3 (Low): Transcript store API is pure-function style and returns `Transcript`
- Impact: future mutable store implementation may need wrappers.
- Remediation: keep this functional contract as canonical and allow mutable internals behind it.
